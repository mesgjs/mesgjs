1. Operator-Style Words

In Part 1 you used messages like (add), (sub), (mul), and (div) to do arithmetic. Mesgjs also provides shorter operator-style words (often called "op-words") that are aliases for those same messages. They look like the symbols you might already know from math:

Long formShort formMeaningExampleResult
(add n)(+ n)Add3(+ 4)7
(sub n)(- n)Subtract10(- 3)7
(mul n)(* n)Multiply3(* 4)12
(div n)(/ n)Divide12(/ 4)3
(pow n)(** n)Power2(** 8)256
(mod n)(/+ n)Remainder10(/+ 3)1

These are just aliases — they do exactly the same thing as the long forms. You can use whichever style you prefer, and you can mix them freely:

// Long form
@c(log "The answer"(join " is:") 6(mul 7)(div 2))

// Short form — identical result
@c(log "The answer"(+ " is:") 6(* 7)(/ 2))

Both lines display: The answer is: 21

Op-words are words in the Mesgjs sense — they're just words that happen to be made entirely of punctuation characters. They follow the same message-sending rules as any other operation name. The + in (+ n) is the operation name, not a traditional infix operator — it still goes inside the parentheses, after the receiver.

The join message on strings also has a short alias: (+). So you can write:

@c(log Hello(+ ", " world "!"))

This displays: Hello, world!

There are also comparison op-words you'll use later when you learn about conditionals:

Long formShort formMeaning
(eq v)(= v)Equal to
(ne v)(!= v)Not equal to
(gt v)(> v)Greater than
(ge v)(>= v)Greater than or equal
(lt v)(< v)Less than
(le v)(<= v)Less than or equal

Exercise 1.1 — Rewrite with Op-Words

Rewrite this expression using op-word aliases instead of the long-form names:

@c(log 10(add 5)(mul 3)(sub 1))
@c(log 10(+ 5)(* 3)(- 1))

Result: 44. Both forms are identical — (10 + 5) × 3 − 1 = 44.

Exercise 1.2 — Powers and Remainders

Try these op-word examples:

@c(log 2(** 10))
@c(log 17(/+ 5))

What do you get? (Hint: 2(** 10) is 2 raised to the power of 10; 17(/+ 5) is the remainder (mnemonic: the additional amount left over after division) when 17 is divided by 5.)

2(** 10) → 1024

17(/+ 5) → 2 (17 = 3×5 + 2)

Exercise 1.3 — String Join with Op-Word

Use the (+) op-word alias for join to build a greeting:

@c(log Good(+ " morning, " world "!"))

Try building your own greeting using (+).

2. Lists

A list is a collection that can hold any number of values — numbers, text, other lists, code blocks, or anything else. Lists are used heavily in Mesgjs.

Positional and named items

A list can contain two kinds of items:

  • Positional items — values listed one after another, automatically numbered starting from 0
  • Named items — values paired with a name using =, just like named values in messages

A list can have both kinds at the same time:

// Three positional items (indices 0, 1, 2)
[red green blue]

// Two named items
[name=Alice age=30]

// Mixed: one positional item and two named items
["software engineer" name=Alice age=30]

Creating a list

You create a list by putting values between [ and ]. Lists can contain a mix of value types, and they can even contain other lists:

// A list of three numbers
#(nset scores=[10 20 30])

// A list with named items
#(nset person=[name=Alice age=30])

// A list containing another list
#(nset breakfast=['orange juice' crepes=[milk eggs flour]])

In the last example, #breakfast is a list with two items:

  • A positional value (index 0): the string 'orange juice'
  • A named value: crepes, which is itself a list of three positional items

Accessing list items

Use (at key) to get an item from a list. You can provide a series of keys to navigate into nested lists in one step:

#(nset breakfast=['orange juice' crepes=[milk eggs flour]])

// Get the first positional item (index 0)
@c(log #(at breakfast 0))

// Get the named item "crepes"
@c(log #(at breakfast crepes))

// Get the first item inside "crepes" (index 0)
@c(log #(at breakfast crepes 0))

The output will be:

  • orange juice
  • The crepes list (displayed as its contents)
  • milk
Notice that #(at breakfast 0) uses the full form of the at message sent directly to #. This is more efficient than the two-step #breakfast(at 0) (which first retrieves the list, then sends at to it), though both work. For deeply nested access, the full form is preferred.

Setting values in lists (including nested lists)

You can set or update a value inside a list — even deep inside a nested list — using the (set) message. Provide the key path leading to the item, then use to= for the new value:

// Start with the breakfast list from before
#(nset breakfast=['orange juice' crepes=[milk eggs flour]])

// Add a "toppings" list inside the nested "crepes" list
#(set breakfast crepes toppings to=[berries 'whipped cream'])

// Verify it was set
@c(log #(at breakfast crepes toppings 0))
@c(log #(at breakfast crepes toppings 1))

Output:

  • berries
  • whipped cream

The key path breakfast crepes toppings tells Mesgjs to navigate into #breakfast, then into the crepes item, and set the toppings key there. Mesgjs creates any missing intermediate lists automatically.

You can also update a single value at any depth the same way:

// Change the first crepe ingredient from "milk" to "oat milk"
#(set breakfast crepes 0 to='oat milk')
@c(log #(at breakfast crepes 0))

This displays: oat milk

The shortcut for named items

Just like #name is a shortcut for #(at name), you can chain the shortcut for named items in a list:

#(nset config=[host=localhost port=8080])

// Full form
@c(log #(at config host))

// Two-step shortcut
@c(log #config(at host))

Both display localhost.

Lists, storage, and message parameters

You may have noticed that the storage objects #, %, !, and others respond to the same (at), (nset), and (set) messages as lists. That's because storage objects are lists — they're just special built-in ones. Everything you learn about lists applies to storage too.

In fact, the values you pass inside a message are also a list — just without the [ and ] brackets. The values start after the operation name and end at the closing ). This is why named values work the same way in messages and in lists — they're the same underlying concept:

// Named values in a message parameter list
#(nset name=Alice age=30)

// Named values in a [ ] list
[name=Alice age=30]

In both cases, name=Alice and age=30 are named items in a list. The with=", " you used in Part 1's join message is a named item in the message's parameter list, just like name=Alice is a named item in a [...] list.

// Add an address nested inside person

3. Code Blocks

A code block is an object that contains a chunk of Mesgjs code. You can store it, pass it around, and run it whenever you want — even multiple times. Code blocks are the foundation for reusable code in Mesgjs.

Creating a code block

A code block is written as Mesgjs code surrounded by curly braces. There are two forms:

FormSyntaxDefault return value
Standard block{ ... }@u (undefined)
Value-returning block{ ... !}The last value in the block
!} is a single token — it means "end this block and return the last value." Do not put a space between ! and }, as that would change the meaning.

Writing a code block creates it as an object — the code inside is not executed at that point. The block just sits there, ready to be used, until something explicitly runs it.

Running a code block

Send the (run) message to a code block to execute it:

// Create a code block and store it
#(nset greet={ @c(log "Hello from a code block!") })

// Run it
#greet(run)

This displays: Hello from a code block!

You can run the same block as many times as you like:

#(nset greet={ @c(log "Hello!") })
#greet(run)
#greet(run)
#greet(run)

This displays Hello! three times.

Value-returning blocks

When a block ends with !}, it returns the last value evaluated inside it. This is useful when you want to use the result of a block:

// This block returns the result of the calculation
#(nset calc={ 6(* 7) !})
@c(log "The result is:" #calc(run))

This displays: The result is: 42

A self-returning block

Here's a clever trick: a block can return itself, which lets you chain (run) calls:

#(nset greet={ @c(log Welcome to Mesgjs) #greet !})
#greet(run)(run)(run)

What's happening step by step:

  1. #(nset greet=...) stores the code block in scratch storage as greet
  2. The code block, when run, logs "Welcome to Mesgjs" and then returns #greet — itself!
  3. #greet(run) runs the block the first time, returning the block itself
  4. (run) is chained onto that result, running it a second time
  5. Another (run) runs it a third time

This displays Welcome to Mesgjs three times.

Code blocks are the building blocks for functions, loops, and conditionals in Mesgjs. You'll learn about those in Part 3. For now, focus on understanding how to create, store, and run them.

Exercise 3.1 — Your First Code Block

Create a code block that displays your name, store it, and run it:

#(nset sayName={ @c(log My name is Alice) })
#sayName(run)

Change the name to your own, then run it three times by calling #sayName(run) three times.

Exercise 3.2 — Value-Returning Block

Create a block that calculates and returns a value, then use the result:

#(nset area={ 5(* 8) !})
@c(log "Area of 5×8 rectangle:" #area(run))

Try changing the dimensions. What happens if you use } instead of !}?

With } (standard block), the block returns @u (undefined) instead of the calculation result. The log would display Area of 5×8 rectangle: undefined (or similar, depending on how the browser displays undefined).

With !} (value-returning block), the last value — the result of 5(* 8) — is returned, so the log displays Area of 5×8 rectangle: 40.

Exercise 3.3 — Block Using Scratch Storage

A code block can read from scratch storage. Try this:

#(nset greeting="Good morning")
#(nset name=Alice)
#(nset sayHi={ @c(log #greeting #name) })
#sayHi(run)

Now change #name to a different value and run the block again. Does it pick up the new value?

#(nset name=Bob)
#sayHi(run)

Yes! The block reads #name at the time it runs, not at the time it was created. So after setting #name to Bob, running the block again displays Good morning Bob.

Exercise 3.4 — Self-Returning Block

Create a countdown block that returns itself, then chain three runs:

#(nset count=3)
#(nset tick={
  @c(log #count)
  #(nset count=#count(- 1))
  #tick
!})
#tick(run)(run)(run)

What do you see? (Don't worry about the #(nset count=...) line inside the block — it updates the scratch value each time the block runs. You'll learn more about this pattern in Part 3.)

The output is:

3
2
1

Each run logs the current count, decrements it by 1, and returns the block itself so the next (run) in the chain can execute it again.

4. Comments

A comment is text in your code that Mesgjs completely ignores. Comments are notes for the people reading the code — including your future self! Good comments explain why something is done, not just what.

Single-line comments

A single-line comment starts with // and continues to the end of that line:

// This entire line is a comment
@c(log Hello) // This part is a comment; the code before it still runs

Everything from // to the end of the line is ignored. You can put a comment on its own line, or at the end of a line of code.

Multiple // markers on the same line don't create separate comments — it's all one comment from the first // to the end of the line:

// <-- starts here // all // one // comment // ends here -->

Multi-line comments

A multi-line comment starts with /* and ends with */. Everything between them is ignored, even if it spans multiple lines:

/* This is a multi-line comment.
   It can span as many lines as you need.
   Mesgjs ignores all of this. */
@c(log This runs normally)

Multi-line comments can also appear within a line:

@c(log /* this part is ignored */ Hello)
Multi-line comments do not nest. The comment ends at the first */ encountered, regardless of any /* inside it: 
/* <-- starts here /* ends here --> */ not part of comment */
In this example, the comment ends at the first */, and the text after it (not part of comment */) would be treated as code — which would likely cause an error. Be careful not to nest /* ... */ comments.

When to use which

  • Use // for short notes on a single line
  • Use /* ... */ for longer explanations, or to temporarily disable a block of code

Putting it all together

Here's a well-commented example using everything from Part 2:

/* Calculate the area and perimeter of a rectangle.
   Width = 8, Height = 5 */

// Store the dimensions
#(nset width=8 height=5)

// Calculate area and perimeter using op-words
#(nset area=#width(* #height))
#(nset perimeter=2(* #width(+ #height)))

// Display the results
@c(log "Width:" #width "Height:" #height)
@c(log "Area:" #area)
@c(log "Perimeter:" #perimeter)

This displays:

Width: 8 Height: 5
Area: 40
Perimeter: 26

One handy use of comments is temporarily disabling a line of code. Try commenting out the second log line:

@c(log Line 1 is active)

Exercise 4.3 — Multi-Line Comment

Use a multi-line comment to add a header block to this program:

#(nset celsius=100)
#(nset fahrenheit=#celsius(* 9)(/ 5)(+ 32))
@c(log #celsius "°C =" #fahrenheit "°F")

Add a /* ... */ comment at the top that describes what the program does.

/* Temperature converter: Celsius to Fahrenheit

#(nset celsius=100)
#(nset fahrenheit=#celsius(* 9)(/ 5)(+ 32))
@c(log #celsius "°C =" #fahrenheit "°F")

5. Summary

Great work! Here's a recap of everything you've learned in Part 2:

ConceptExampleWhat it does
Op-word alias6(* 7)(/ 2)Same as 6(mul 7)(div 2) — gives 21
String join op-wordHello(+ " world")Same as Hello(join " world")
Create a list[10 20 30]A list of three numbers
Named item in list[name=Alice age=30]A list with two named items
Access by index#(at scores 0)Gets the first item in the scores list
Access by name#(at person name)Gets the name item from the person list
Nested access#(at data scores 1)Gets index 1 of the scores list inside data
Set nested value#(set data scores 1 to=99)Sets index 1 of the scores list inside data to 99
Standard code block{ @c(log hello) }A block that returns @u (undefined) when run
Value-returning block{ 6(* 7) !}A block that returns its last value (42) when run
Run a block#myBlock(run)Executes the stored code block
Single-line comment// note hereIgnored from // to end of line
Multi-line comment/* note here */Ignored from /* to first */

Key things to remember

  • Op-words like *, /, +, - are just aliases — they work exactly like mul, div, add, sub
  • Lists are created with [ and ]; items can be positional (numbered from 0) or named
  • Storage objects like # are lists too — the same (at), (nset), and (set) messages work on both
  • Message parameters are also a list — the values between the operation name and the closing ) work just like [...]
  • { ... } returns @u; { ... !} returns the last value — and !} is a single token (no space!)
  • Code blocks run when sent (run); they can be stored, reused, and chained
  • // comments end at the line break; /* */ comments end at the first */ and do not nest

Final Exercise — Putting It All Together

Write a Mesgjs program that:

  1. Uses a multi-line comment to describe what the program does
  2. Stores a list of three product prices
  3. Creates a code block that calculates and displays the total of the three prices (using op-words)
  4. Runs the code block
/* Shopping cart total calculator.
   Adds up three product prices and displays the total. */

// Store the prices
#(nset prices=[12 25 8])

// Code block to calculate and display the total
#(nset showTotal={
  // Add all three prices together
  #(nset total=#(at prices 0)(+ #(at prices 1))(+ #(at prices 2)))
  @c(log "Total:" #total)
})

// Run it
#showTotal(run)

Result: Total: 45
