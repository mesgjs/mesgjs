1. More Storage Types

In Part 1 you learned about # (scratch storage) – a temporary notepad for values during the current operation. Mesgjs has several other storage objects, each with a different scope (what it's used for) and lifetime (when and how long it's available). Here's the full picture:

StorageNameScope / Lifetime
#ScratchTemporary; per dispatch (current operation)
!Message parametersThe values passed in the current message call
%Object propertiesPersistent; belongs to the current object
%*Global shared storagePersistent; shared across the entire program
%/Module storagePersistent; private to the current module

All of these storage objects respond to the same messages you already know: (at), (nset), (set), and so on – because they're all lists under the hood.

Message parameters: !

The ! storage object holds the values passed in the current message call. You'll use it inside functions (covered below) and object interface methods (covered in Part 4).

Object properties: %

The % storage object holds values that belong to the current object and persist across multiple calls. This is how objects remember information. You'll use % most when creating your own "interfaces" (covered in Part 4), but you can also use it in functions to give them persistent memory – more on that below.

Global shared storage: %*

The %* storage object is shared across the entire program. Values stored here are accessible from anywhere. It works just like #, but the values persist for the lifetime of the program rather than just the current operation:

// Store a value in global storage
%*(nset appName="My Mesgjs App" version=1)

// Access it from anywhere
@c(log %*appName "version" %*version)

Output: My Mesgjs App version 1

In the examples throughout this tutorial, you'll often see functions stored in %* (global storage). This is the idiomatic Mesgjs pattern for defining reusable functions – it makes them accessible from anywhere in the program. You'll see this in action starting in Section 3.

Exercise 1.1 – Global Storage

Store your name and favorite color in global storage, then display them:

%*(nset myName=Alice favoriteColor=blue)
@c(log %*myName "likes the color" %*favoriteColor)

Change the values to your own name and color. Then try updating just the color using %*(nset favoriteColor=green) and displaying it again.

2. Safe Access: ? Variants and Default Values

What happens if you try to access a value that doesn't exist? By default, Mesgjs generates an error. But often you'd rather get a safe fallback instead. Mesgjs gives you two tools for this.

The ? shortcut variants

The simplest approach is to use a ? shortcut variant – a way of telling Mesgjs "I know this might not have a defined value yet, and that's OK." These work just like the regular shortcuts (#name, %name, etc.) but return @u (undefined) instead of generating an error when the key is missing:

Regular shortcutSafe shortcutBehavior when missing
#name#?nameReturns @u
%name%?nameReturns @u
!name!?nameReturns @u
%*name%*?nameReturns @u
%/name%/?nameReturns @u
// When #score doesn't exist yet
@c(log #?score)   // @u (undefined) – no error

#(nset score=95)
@c(log #?score)   // 95

Output (you may have to reload the REPL page in your browser):

undefined
95

The else parameter

Sometimes you want a more specific default value than @u. The (at) message accepts an else parameter. If the key doesn't exist (or any of the keys, when there's more than one), the else value is returned instead of generating an error:

// When #score doesn't exist yet
@c(log #(at score else=0))

// Now set it
#(nset score=95)
@c(log #(at score else=0))

Output:

0
95

The else value can be any expression, including a code block that will be run (if the value is needed) to produce the default:

@c(log #(at name else={ -(joining U n k n o w n) !}))

Output: U-n-k-n-o-w-n

Just as the shortcut #name (for example) sends the message #(at name) for you, #?name sends the message #(at name else=@u) for you.

Exercise 2.1 – Safe Access with ?

Try accessing a scratch value that doesn't exist using both the regular and safe shortcuts:

// This will generate an error:
// @c(log #missing)

// This returns @u safely:
@c(log #?missing)

// Now set it and try again:
#(nset missing=found)
@c(log #?missing)

What do you see for each line?

  • Line 1 (commented out): would generate an error
  • Line 2: displays undefined
  • Line 3: displays found

Exercise 2.2 – Default Values with else

Use the else parameter to provide a specific default when a value might not exist. Try this pattern for a "visit counter" that starts from zero:

// First visit: #visits doesn't exist yet, so else=0 is used
#(nset visits=#(at visits else=0)(+ 1))
@c(log "Visit count:" #visits)

// Second visit: #visits now exists

What do you see? Try running it a few more times by adding more lines.

3. Functions

In Part 2 you learned about code blocks – chunks of code you can store and run. Functions are a step up from code blocks: they have their own persistent storage, fresh scratch storage every call, and they can accept parameters.

Creating a function

You create a function by writing a code block and sending it the (fn) message:

// A simple function that squares a number
%*(nset square={ !0(* !0) !}(fn))

The (fn) message creates a new function object. It shares the same code as the original code block, but it is a new and otherwise independent object. Like the original code block, the code inside is not executed yet – it's just stored, ready to be called.

You can also send a (fn) message to a existing function, and it works exactly the same way as if you had sent the message to the original code block.

Calling a function and accessing parameters with !

While code blocks are run with (run), functions are called with (call). The key difference is that (call) accepts parameters – the values you pass become the function's message parameters, accessible inside the function via the ! storage object. Positional parameters are accessed by index: !0 is the first, !1 is the second, and so on. Named parameters are accessed by name:

%*(nset square={ !0(* !0) !}(fn))

@c(log %*square(call 5))    // 25
@c(log %*square(call 12))   // 144

Output:

25
144

Functions vs. code blocks

The key difference between a function and a plain code block is context isolation:

Code block ({ })Function ({ }(fn))
Runs inThe defining dispatch contextA fresh dispatch context
Scratch storage (#)From definition siteFresh, empty each call
Parameters (!)From definition siteValues passed to (call)
Persistent storage (%)From definition siteFrom (fn) parameters
Invoked via(run)(call)

(We'll explain dispatch contexts in Part 4.)

You can think of a code block like an employee working for their employer – it uses the employer's resources and workplace (the defining context). A function is more like an onsite independent contractor – it brings its own tools and allocates fresh resources for each job.

A practical example: temperature conversion

// Convert Celsius to Fahrenheit: F = (C × 9/5) + 32
%*(nset cToF={ !0(* 9)(/ 5)(+ 32) !}(fn))

@c(log "0°C =" %*cToF(call 0) "°F")
@c(log "100°C =" %*cToF(call 100) "°F")
@c(log "37°C =" %*cToF(call 37) "°F")

Output:

0°C = 32 °F
100°C = 212 °F
37°C = 98.6 °F

Named parameters

You can also pass named parameters to a function. Inside the function, access them by name using !:

#(nset introduce={
    @c(log My name is !name and I am !age years old.)
}(fn))

#introduce(call name=Alice age=30)

Output: My name is Alice and I am 30 years old.

Functions calling other functions

Functions can call other functions. Here's an example where one conversion function builds on another:

// Celsius-to-Kelvin converter: K = C + 273.15
%*(nset cToK={ !0(+ 273.15) !}(fn))

// Kelvin-to-Fahrenheit converter: F = (K × 9/5) - 459.67
%*(nset kToF={ !0(* 9)(/ 5)(- 459.67) !}(fn))

// Celsius-to-Fahrenheit converter using the two other functions
%*(nset cToF={
    #(nset kelvin=%*cToK(call !0))
    %*kToF(call #kelvin)
    // alternatively: %*kToF(call %*cToK(call !0))
!}(fn))

@c(log "0°C =" %*cToF(call 0) "°F")
@c(log "100°C =" %*cToF(call 100) "°F")

Output:

0°C = 32 °F
100°C = 212 °F

The cToF function calls cToK first to convert to Kelvin, then calls kToF to convert that result to Fahrenheit.

Default parameter values with else

The else parameter you learned in Section 2 is especially useful inside functions, where a caller might or might not pass a particular argument:

#(nset greet={
    // Use the provided name, or "stranger" if none was given
    #(nset name=!(at 0 else=stranger))
    @c(log Hello, #name !)
!}(fn))

#greet(call Alice)
#greet(call)

Output:

Hello, Alice !
Hello, stranger !

Persistent state with % and else

Another common pattern is using else with % (object properties) to initialize a value the first time it's needed. Here's a counter function that starts from zero automatically:

// A counter function that remembers how many times it's been called
#(nset counter={
    %(nset count=%[count else=0](+ 1))
    @c(log Count: %count)
}(fn))

#counter(call)
#counter(call)
#counter(call)

Output:

Count: 1
Count: 2
Count: 3

On the first call, %count doesn't exist yet, so %[count else=0] returns 0 (instead of generating an error). Adding 1 gives 1, which is saved back to %count. On subsequent calls, %count already exists, so its current value is used.

Setting % at creation time with (fn) parameters

The else pattern above initializes % the first time the function runs. But you can also set % values when the function is created, by passing parameters to (fn). Those parameters become the function's initial persistent state – already set before the function is ever called.

Here's a code block that multiplies its input by a factor stored in %:

// Save the code block itself (not a function yet)
#(nset multiplyBlock={ !0(* %factor) !})

// Create two separate functions from the same code block,
// each with a different starting value for %factor
#(nset timesTwo=#multiplyBlock(fn factor=2))
#(nset timesTen=#multiplyBlock(fn factor=10))

@c(log #timesTwo(call 6))    // 12
@c(log #timesTen(call 6))    // 60

Output:

12
60

#multiplyBlock(fn factor=2) creates a function whose %factor is already set to 2. Sending (fn factor=10) to the same code block creates a completely independent function with its own %factor of 10. The two functions share the same code but have separate persistent state.

This is the idiomatic Mesgjs pattern for giving a function access to a value that is known at creation time. It is generally preferred over storing such values in %* (global storage) or %/ (module storage), since it keeps each function self-contained.

Exercise 3.1 – Your First Function

Write a function that takes a number and returns its cube (number × number × number):

%*(nset cube={ !0(* !0)(* !0) /* or just !0((* !0 !0)) */ !}(fn))

@c(log %*cube(call 3))   // 27
@c(log %*cube(call 5))   // 125

Exercise 3.2 – Function with Multiple Parameters

Write a function that takes a name and an age, and returns a greeting string like "Alice is 30 years old":

%*(nset describe={
    !0(+ " is " !1 " years old")
!}(fn))

@c(log %*describe(call Alice 30))
@c(log %*describe(call Bob 25))

Exercise 3.3 – Persistent State with %

Create a function that accumulates a running total. Each call adds a number to the total and returns the new total:

#(nset accumulate={ %(nset total=%[total else=0](+ !0)) %total !}(fn))

@c(log #accumulate(call 10))   // 10
@c(log #accumulate(call 5))    // 15
@c(log #accumulate(call 20))   // 35

Each call adds to %total, which persists between calls because it's stored in the function's own % storage.

4. Conditionals

Conditionals let your program make decisions – doing different things depending on whether something is true or false. Mesgjs provides two conditional operations: (if) and (case).

The (if) operation

The (if) operation is sent to @c (the core object). It takes pairs of test/action values, and runs the action for the first test that is true:

@c(if
    test1 action1
    test2 action2
    ...
    else=defaultAction
)

Each test and action can be a code block, an expression, or a simple value. Tests are evaluated one at a time; the first one that returns a truthy value causes its corresponding action to run. If no test is true, the else action runs (or @u is returned if there's no else).

The first condition is typically not a block (since it's always evaluated, there's no benefit to lazy evaluation). Subsequent conditions are usually blocks so they're only evaluated when needed.

Stylistically, when there's only one condition, it can go on the same line as (if). When there are multiple conditions, it's recommended to put the first condition on the next line:

// Single condition – compact form is fine
@c(if cond1 { then1 }
   else={ elseBlock })

// Multiple conditions – more readable to start on a new line
@c(if
    cond1 { then1 }
    { cond2 !} { then2 }
    else={ elseBlock }
)

Here's a simple example:

#(nset x=7)

@c(if 
    #x(> 10) { @c(log "x is greater than 10") }
    { #x(> 5) !}  { @c(log "x is greater than 5") }
    else={ @c(log "x is 5 or less") }
)

Output: x is greater than 5

The tests are checked in order. Since 7 > 10 is false, the first action is skipped. Since 7 > 5 is true, the second action runs.

The test blocks use !} (value-returning blocks) because their result – true or false – needs to be returned to (if). The action blocks use } (standard blocks) when they don't need to return a value, or !} when they do.

Simple one-branch (if)

If you only need to do something when a condition is true (and nothing otherwise), you can use a single test/action pair without else:

#(nset temperature=35)

@c(if #temperature(> 30) {
    @c(log "It's hot outside!")
})

Output: It's hot outside!

Using (if) to return a value

The (if) operation returns the result of whichever action ran. This lets you use it as an expression:

#(nset score=85)

#(nset grade=@c(if
    #score(>= 90) A
    { #score(>= 80) !} B
    { #score(>= 70) !} C
    else=F
))

@c(log "Grade:" #grade)

Output: Grade: B

Here the actions are plain values (A, B, C, F) rather than code blocks. When an action is a plain value (not a code block), (if) returns it directly.

The (case) operation

When you want to compare one value against several possibilities, (case) is cleaner than a chain of (if) tests. It takes a reference value, then pairs of comparison/result values:

#(nset day=3)

#(nset dayName=@c(case #day
    1 Monday
    2 Tuesday
    3 Wednesday
    4 Thursday
    5 Friday
    else=Weekend
))

@c(log #dayName)

Output: Wednesday

(case) compares the first value (#day) to each comparison value in order. When it finds a match, it returns the corresponding result. If nothing matches, it returns the else value (or @u).

Logical operations: (and), (or), (not)

You can combine conditions using logical operations on @c:

OperationShort formMeaning
@c(and a b)@c(& a b)True if both a and b are true
@c(or a b)@c(| a b)True if either a or b is true
@c(not a)@c(~ a)True if a is false
#(nset age=25 hasTicket=@t)

@c(if @c(and #age(>= 18) #hasTicket) { @c(log "You may enter.") }
else={ @c(log "Entry denied.") })

Output: You may enter.

The and and or operations always need the first value (the a in the table), but only evaluate any subsequent code-block values (i.e. for calculated values) when they might affect the result.

    #n(> 0) { @c(log positive) }

%*(nset getSeason={

        12 Winter  1 Winter  2 Winter

Exercise 4.4 – Logical Combinations

Write a function that checks if a year is a leap year. A year is a leap year if it's evenly divisible by 400, or if it's evenly divisible by 4, but not by 100.

%*(nset isLeapYear={
    @c(if
        !0(/+ 400)(= 0) @t
        { @c(and !0(/+ 4)(= 0) { !0(/+ 100)(!= 0) !}) !} @t
		// Alternatively:
		// { !0(/+ 100)(= 0) !} @f
		// { !0(/+ 40)(= 0) !} @t
        else=@f
    )
!}(fn))

@c(log %*isLeapYear(call 2000))   // @t (divisible by 400)
@c(log %*isLeapYear(call 1900))   // @f (divisible by 100 but not 400)
@c(log %*isLeapYear(call 2024))   // @t (divisible by 4)
@c(log %*isLeapYear(call 2023))   // @f

Advanced Notes – Conditionals

  • Conditionals are library features, not language syntax. Unlike most programming languages, (if) and (case) (as well as the looping capabilities we'll cover below) aren't actually built into the Mesgjs language itself – they're part of the standard library that normally comes with Mesgjs. It's possible to add additional conditional or looping capabilities, or even to replace the standard ones if there's a reason to do so.

5. Early Returns

Sometimes you want to exit a function (or code block) before reaching the end – for example, when an error condition is detected, or when you've found the answer you were looking for. Mesgjs provides @d(return) for this purpose.

The @d(return) message

@d is the current dispatch object – it represents the currently-executing message and various details about it (context). Sending it the (return) message immediately exits the current function or object message, optionally returning a value:

@d(return)           // Exit with no value (@u)
@d(return someValue) // Exit and return someValue

Here's a practical example – a function that validates input and returns early if something is wrong:

%*(nset safeDivide={
    // Return @u if dividing by zero
    @c(if !1(= 0) { @d(return) })
    !0(/ !1)
!}(fn))

@c(log %*safeDivide(call 10 2))    // 5
@c(log %*safeDivide(call 10 0))    // undefined (early return)

Output:

5
undefined

When !1 (the divisor) is 0, the function returns immediately with @u. Otherwise, it continues and returns the division result.

Returning a specific value early

%*(nset classify={
    @c(if
        !0(< 0) { @d(return negative) }
        { !0(= 0) !} { @d(return zero) }
    )
    positive
!}(fn))

@c(log %*classify(call -5))   // negative
@c(log %*classify(call 0))    // zero
@c(log %*classify(call 7))    // positive

Output:

negative
zero
positive

Each early return exits the function immediately with the specified value. The last line (positive) is only reached if neither early return was triggered.

This example is a bit contrived, since @c(if) with two conditions and an else would work cleanly here, but if the default handling requires more complicated code, this pattern can help you reduce the number and/or depth of nested blocks.

Early returns are especially useful for "guard clauses" – checks at the top of a function that handle edge cases before the main logic runs. This keeps the main logic clean and unindented.

Exercise 5.1 – Guard Clause

Write a function that computes the square root of a number, but returns @u early if the number is negative (since square roots of negative numbers are not real):

%*(nset safeSqrt={
    @c(if !0(< 0) { @d(return) })
    !0(sqrt)
!}(fn))

@c(log %*safeSqrt(call 25))    // 5
@c(log %*safeSqrt(call -4))    // undefined

Exercise 5.2 – Early Return with a Value

Write a function that accepts three numbers and returns the first one that is negative, or @u if they're all non-negative.

%*(nset firstNegative={
    @c(if
        !0(< 0) { @d(return !0) }
        { !1(< 0) !} { @d(return !1) }
        { !2(< 0) !} { @d(return !2) }
    )
!}(fn))

@c(log %*firstNegative(call 3 -1 5))    // -1
@c(log %*firstNegative(call 1 2 3))     // undefined

6. Loops

Loops let you repeat an action multiple times without writing the same code over and over. Mesgjs provides loops through the @loop interface. You create a loop object with @c(get @loop), then send it messages to run iterations.

Creating a loop object

#(nset loop=@c(get @loop))

A loop object is a reusable tool. You can run it multiple times (though not simultaneously).

Fixed-count loops: (run)

The (run) message runs a code block a fixed number of times:

#(nset loop=@c(get @loop))

#loop(run { @c(log Hello!) } times=3)

Output:

Hello!
Hello!
Hello!

Loop counters

Inside the loop body, you can ask the loop object for the current iteration number:

MessageMeaning
loop(num)0-based iteration number (0 on first iteration)
loop(num1)1-based iteration number (1 on first iteration)
loop(rem)Iterations remaining after this one
loop(times)Total planned iterations
#(nset loop=@c(get @loop))

#loop(run {
    @c(log Iteration #loop(num1) of #loop(times))
} times=4)

Output:

Iteration 1 of 4
Iteration 2 of 4
Iteration 3 of 4
Iteration 4 of 4

Building a sum with a loop

// Sum 1 + 2 + 3 + ... + 10
#(nset loop=@c(get @loop) total=0)

#loop(run {
    #(nset total=#total(+ #loop(num1)))
} times=10)

@c(log "Sum 1..10 =" #total)

Output: Sum 1..10 = 55

Conditional loops: (while)

The (while) message runs a code block repeatedly as long as one or more conditions remain true. Typically, a pre-condition is a code block passed as the pre parameter, and the body is the second argument:

#(nset loop=@c(get @loop) n=1)

#loop(while pre={ #n(<= 5) !} { @c(log #n) #(nset n=#n(+ 1)) })

Output:

1
2
3
4
5

The pre test block is evaluated before each iteration. When it returns false (or any falsy value), the loop stops.

The pre test uses a value-returning block (!}) because its result – true or false – needs to be returned, per iteration, to the loop. In this case, the body block uses a standard block (}) since its return value isn't needed here.
Sometimes you want a loop to execute at least once before testing whether it should continue to execute. In that case, use post as the name for your test block, and it will be checked after each loop iteration instead of before. By convention, the post test is placed after the body block, so that the visual arrangement matches the order in which the blocks are run.

Stopping a loop early: (stop)

You can exit a loop before it finishes using loop(stop):

// Find the first number divisible by both 3 and 7
#(nset loop=@c(get @loop) found=@u)

#loop(run {
    #(nset n=#loop(num1))
    @c(if @c(and #n(/+ 3)(= 0) { #n(/+ 7)(= 0) !}) {
        #(nset found=#n)
        #loop(stop)
    })
} times=100)

@c(log "First number divisible by 3 and 7:" #found)

Output: First number divisible by 3 and 7: 21

Collecting loop results

If you pass collect=@t to (run) or (while), the loop collects all the body's return values into a list:

#(nset loop=@c(get @loop))

// Build a list of squares: [1, 4, 9, 16, 25]
#(nset squares=#loop(run {
    #loop(num1)(* #loop(num1))
!} times=5 collect=@t))

@c(log #squares(toString))

Output: [(1 4 9 16 25)]

Iterating over a list

While you could iterate over the positional values of a list using a @loop with manual index arithmetic, Mesgjs provides a purpose-built tool for this: the @kvIter (key-value iterator) interface. It is cleaner, handles both index and named keys automatically, and supports the same collect=@t option as @loop.

You create an iterator with @c(get @kvIter), and then call (for source {...}) to run a code block for each item in the list.

Inside the code block, the iterator object gives you three messages:

MessageReturns
iter(key)The current key (a number for index entries, a string for named entries)
iter(value)The current value
iter(isIndex)@t if the key is an index (non-negative integer), @f if it's a named key

Handling index entries (positional values) and named entries together

#(nset myList=[first middle=second last])
#(nset iter=@c(get @kvIter))

#iter(for #myList {
    @c(log #iter(key) : #iter(value) 'index?' #iter(isIndex))
})

Output:

0 : first index? false
middle : second index? true
1 : last index? false

As you can see, when you pass an unnamed code block, it is called for every list item (indexed/positional or named value).

Handling mixed lists with index= and named=

Sometimes you only care about one type of value, or you care about both types but want to handle each type differently. You could use the (isIndex) message to handle this, but the (for) message offers an easier alternative, letting you provide separate handler blocks for each kind:

#(nset mixed=[greeting=hello 42 label=world 7])
#(nset iter=@c(get @kvIter))

#iter(for #mixed
    index={ @c(log index #iter(key) '=' #iter(value)) }
    named={ @c(log named #iter(key) '=' #iter(value)) }
)

Output:

named greeting = hello
index 0 = 42
named label = world
index 1 = 7

Collecting results with @kvIter

Just like @loop, you can pass collect=@t to gather all the iteration results into a list:

#(nset numbers=[3 1 4 1 5] iter=@c(get @kvIter))

// Build a list of doubled values
#(nset doubled=#iter(for #numbers {
    #iter(value)(* 2)
!} collect=@t))

@c(log #doubled(toString))

Output: [(6 2 8 2 10)]

Stopping early and the else block

You can stop iteration early with iter(stop), and you can provide an else block that runs (and whose value is returned) when the source list is empty:

#(nset iter=@c(get @kvIter))

// Find the first value greater than 3
#(nset result=#iter(for [1 2 3 4 5] {
    @c(if #iter(value)(> 3) { #iter(stop result=#iter(value)) })
!} else=@u))

@c(log "First value > 3:" #result)

Output: First value > 3: 4

Use @kvIter when you need to work with the keys and values of a list, or when you want clean handling of mixed index/named lists. Use @loop when you need a fixed count ((run times=n)) or a condition-based loop ((while)) that isn't tied to a specific data source.

7. Summary

Excellent work! Here's a recap of everything you've learned in Part 3:

ConceptExampleWhat it does
Message parameters!0, !nameFirst positional param; named param
Object properties%(nset x=1), %xPersistent storage for the current object
Global storage%*(nset f=...), %*fShared across the entire program
Safe access#?nameReturns @u if key doesn't exist
Default value#(at x else=0)
#[x else=0]		Returns 0 if x doesn't exist
Create a function{ !0(* 2) !}(fn)Creates a function from a code block
Pre-initialize function stateblock(fn x=1 y=2)Creates a function with %x and %y already set
Call a function%*double(call 5)Calls the function with parameter 5
Conditional@c(if test action else=default)Runs action if test is true
Case@c(case value 1 one 2 two else=other)Matches value against cases
Early return@d(return value)Exits current function/message with value
Fixed looploop(run { ... } times=5)Runs the block 5 times
While looploop(while pre={ test !} { body })Runs body while test is true
Stop looploop(stop)Exits the loop immediately
Collect resultsloop(run { value !} times=n collect=@t)Returns a list of all block results
List iterator@c(get @kvIter)Creates a key-value iterator for a list
Iterate forwarditer(for myList { iter(key) iter(value) })Visits every list item in order
Iterate by key typeiter(for myList index={...} named={...})Separate blocks for index vs. named entries

Key things to remember

  • Functions are created with { ... }(fn) and called with (call params)
  • Inside a function, ! holds the call parameters; % holds persistent state; # is fresh scratch storage
  • #?name and ![key else=default] (or !(at key else=default)) are your tools for safe, default-value access
  • @c(if) takes test/action pairs; @c(case) matches a value against alternatives
  • @d(return value) exits the current function or message immediately
  • Loops require a loop object: @c(get @loop)
  • loop(num) is 0-based; loop(num1) is 1-based – use num for list indexing, num1 for display
  • To iterate (loop) over a list's items, use @kvIter: create with @c(get @kvIter), then call iter(for source { ... }); inside the block, iter(key) and iter(value) give you the current item key and value
